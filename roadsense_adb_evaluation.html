What's strong

✓Dual-pipeline design (4-component + 3-module) is genuinely innovative — running two independent methodologies gives built-in internal validation, which is exactly what judges want from "methodological robustness."
✓Module C with CLIP fallback for street-level imagery is a smart hedge — if Mapillary coverage is thin (common in rural Asia-Pacific), CLIP zero-shot still gets signal from available images.
✓OSM enrichment paths (both pyrosm and Overpass API) make scalability to other Asia-Pacific countries credible without needing proprietary data.
✓Dependencies are clean and pinned. ESDA + libpysal for spatial autocorrelation is a well-chosen evaluation tool — Moran's I directly answers the "is your score spatially coherent?" reviewer question.
✓Makefile, pyproject.toml, .env.example, logging, and test scaffold are all present. The repo signals serious engineering hygiene to reviewers.
✓Interactive Folium map output is the right default — judges get a working URL without needing GIS software.

Critical gaps to fix before June 25

blocking Data citations section is blank
The README literally says "Add the exact citations for each project dataset here." This will signal to judges that the submission isn't complete. Fill in every source: the ADB NDA dataset, OSMnx derivation, Mapillary ML tags, and whichever iRAP/WHO reference speed tables you used for Safe System thresholds.
blocking No scored output files in the repo
The README describes output paths (outputs/all_segments_scored.gpkg, outputs/geojson/all_segments_scored.geojson, outputs/maps/roadsense_priority_map.html), but none of these appear to exist in the committed tree. Judges need a working URL for the interactive map and at least sample GeoJSON. Even a 500-segment demo run would validate the pipeline is actually functional.
blocking README run commands don't match package structure
The README says run python -m src.preprocessing, python -m src.module_a, etc. But the actual package is at src/roadsense/ and the entrypoint is run_pipeline.py (which imports from roadsense.pipeline). These contradictory instructions will confuse anyone trying to reproduce the work, and reproducibility is explicitly 25% of the score.
important evaluation_results.md doesn't appear populated
The README lists docs/evaluation_results.md as a key output, but with only 2 commits the pipeline has likely never been run on the actual ADB data. This document needs real numbers: MAE or RMSE on benchmark segments, Moran's I value, sensitivity analysis results. "Your model should have a clear evaluation methodology" is a submission requirement.
important Safe System reference speed thresholds need explicit sourcing
Module A compares against "Safe System reference speeds" but the specific speed values (e.g. 30 km/h near pedestrians, 50 km/h on urban arterials) need to be explicitly cited — probably iRAP or WHO Speed Management Manual — and that mapping committed to a config file so judges can see exactly what you assumed.
important Scoring weights config path mismatch
README says config/scoring_weights.yaml, run_pipeline.py references configs/scoring_weights.yaml (plural). The configs/ directory exists in the tree. Make sure the YAML is actually committed there with real weight values documented and justified.
polish Minor: missing import os in run_pipeline.py
os.chdir(out_dir) is called in the --serve branch but os is never imported. Won't cause issues unless someone tests the serve flag, but it'll fail loudly if they do.
polish Minor: gpd used but not imported in run_pipeline.py
Line ~100: df_scored = gpd.GeoDataFrame(...) in the pyrosm enrichment branch. geopandas as gpd is not imported at the top of that file. Will raise a NameError when the --pbf flag is used.
