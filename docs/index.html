Speed limits should save lives.

RoadSense uses GPS probe data, road network geometry, and street-level imagery to determine whether posted limits protect road users—or are just numbers on a sign.

See the results How it works

Three signals, one score

Every road segment gets a Speed Safety Score (0–1) based on how far the posted limit and actual traffic deviate from Safe System thresholds, how many vulnerable people are nearby, and what the physical road environment looks like.

↑

Speed alignment

Compares posted speed limits and 85th percentile operating speeds against WHO/ITF Safe System reference speeds for each road class and land use context.

☶

VRU exposure

Estimates pedestrian fatality risk using the ITF logistic curve, adjusted by road class, land use, and proximity to schools, markets, and high-ptw zones.

☀

Road environment

Evaluates protective infrastructure using Mapillary detections or a CLIP zero-shot fallback—critical for countries with sparse street-level imagery.

Two independent scoring engines

Every segment is scored twice using different methodologies. The two approaches produce correlated rankings (r > 0.85), confirming the relative risk ordering is robust to weighting choices.

Score = 0.30L + 0.30O + 0.25V + 0.15T. Safe System limits drawn from WHO 2021 and ITF/OECD 2018 per road class and land use pair.

3-Module RoadSense

Module A — Speed alignment35%
Module B — VRU exposure35%
Module C — Road environment30%

SSS = 0.35A + 0.35B + 0.30C. Each module operates independently on a different safety dimension.

Built to replicate

The methodology works in any country with basic road network data—no proprietary data or expensive survey equipment required.

🌎

Any country

OSM road network and Overpass API provide global coverage. The CLIP imagery fallback works even where Mapillary has no data.

📈

Minimal inputs

Minimum viable deployment: OSM roads + speed limit tags + 50 spot-speed measurements. No crash history needed—this is a preventive model.

📖

Open source

MIT-licensed. Config-driven scoring weights, Makefile automation, CI via GitHub Actions. Run it quarterly to track changes.
